🟠 Chapitre 1

HTML5 - Structure & Sémantique

Le HTML5 est le langage de balisage qui structure le contenu d'une page web. La sémantique, c'est choisir la bonne balise pour le bon contenu - ce qui impacte directement le SEO, l'accessibilité et la maintenabilité.

Structure de base d'une page HTML5

Toute page doit commencer par ces éléments fondamentaux :

  • <!DOCTYPE html> - Déclaration du type de document (HTML5)
  • <html lang="fr"> - Racine, avec l'attribut lang obligatoire pour l'accessibilité
  • <head> - Métadonnées (charset, viewport, titre, CSS…)
  • <meta charset="UTF-8"> - Encodage universel des caractères
  • <meta name="viewport" content="width=device-width, initial-scale=1.0"> - Indispensable pour le responsive
  • <body> - Contenu visible de la page
⚠ Piège fréquent

Oublier lang="fr" sur la balise <html> est une erreur d'accessibilité : les lecteurs d'écran ne savent plus dans quelle langue prononcer le contenu.

Hiérarchie des titres - Règle d'or

La hiérarchie des titres doit être logique et séquentielle. Ne pas sauter de niveaux !

  • <h1> - Un seul par page. Titre principal du contenu.
  • <h2> - Sections principales (apparaissent dans la sidebar)
  • <h3> - Sous-sections d'un h2
  • <h4> - Rarement nécessaire. Réfléchir à restructurer.
✗ À ne jamais faire

Utiliser <h3> après <h1> en sautant <h2>. Ou pire : utiliser les balises de titre juste pour leur taille - c'est le rôle du CSS !

Formulaires accessibles

Un formulaire bien structuré doit toujours associer chaque champ à son libellé :

  • <label for="id"> lié à <input id="id"> - Association explicite
  • required, type, autocomplete - Attributs HTML5 natifs
  • aria-describedby - Pour associer un message d'aide ou d'erreur
  • aria-invalid="true" - Signaler une erreur de validation
  • <fieldset> + <legend> - Pour grouper des champs liés
Quelle différence entre <section> et <div> ?
<section> est sémantique : elle représente une partie thématique du contenu et doit avoir un titre (h2, h3…). <div> est un conteneur générique sans signification sémantique - à réserver pour les besoins purement de mise en page.
Peut-on avoir plusieurs <header> dans une page ?
Oui ! <header> peut être utilisé dans chaque <article> ou <section>. Il représente l'en-tête de l'élément parent, pas forcément de la page entière. En revanche, <main> ne doit apparaître qu'une seule fois.
L'attribut alt est-il toujours obligatoire sur les images ?
L'attribut alt est toujours présent, mais il peut être vide (alt="") si l'image est purement décorative. Dans ce cas, le lecteur d'écran l'ignorera complètement. S'il est absent, le lecteur d'écran lira le nom du fichier - ce qui est inutile et perturbant.

🔵 Chapitre 2

CSS3 - Mise en page & Design

Le CSS3 contrôle la présentation visuelle. Maîtriser la cascade, la spécificité, Flexbox et Grid est indispensable pour créer des interfaces modernes et maintenables.

La cascade et la spécificité

Quand plusieurs règles s'appliquent au même élément, le CSS choisit selon :

SélecteurSpécificitéExemple
Élément0-0-1p { }
Classe, attribut, pseudo-classe0-1-0.btn { }
ID1-0-0#header { }
Style inline1-0-0-0style="color:red"
!importantPriorité absolueÀ éviter !
💡 Bonne pratique

Préférer les classes aux IDs en CSS. Éviter !important sauf en dernier recours. Structurer vos sélecteurs pour garder une spécificité basse.

Variables CSS (Custom Properties)

Les variables CSS permettent de centraliser les valeurs réutilisables et de faciliter la maintenance :

  • Déclaration : :root { --couleur-primaire: #1a5fa8; }
  • Utilisation : color: var(--couleur-primaire);
  • Valeur par défaut : var(--ma-var, #fallback)
  • Modifiables dynamiquement avec JavaScript : el.style.setProperty('--var', val)

Flexbox - Mise en page 1D

Flexbox est conçu pour aligner des éléments sur un seul axe (ligne ou colonne).

CSS Grid - Mise en page 2D

CSS Grid est conçu pour des mises en page bi-dimensionnelles (lignes ET colonnes).

  • display: grid - Active le Grid
  • grid-template-columns: repeat(3, 1fr) - 3 colonnes égales
  • grid-template-columns: 260px 1fr - Sidebar + contenu
  • grid-template-columns: repeat(auto-fill, minmax(280px, 1fr)) - Grille responsive automatique
  • gap: 1.5rem - Espacement entre cellules
  • grid-column: 1 / -1 - L'élément s'étend sur toute la largeur
ℹ Flex vs Grid : quand utiliser lequel ?

Flexbox : navigation, boutons côte à côte, centrer un élément, layout 1D.
Grid : layout de page complet, galeries, grilles de cards, layout 2D.

Transitions et animations

  • transition: all 0.3s ease - Transition sur toutes les propriétés
  • transition: color 0.2s, transform 0.3s cubic-bezier(0.4,0,0.2,1) - Cibler des propriétés spécifiques
  • @keyframes nomAnimation { from { } to { } } - Définir une animation
  • animation: nomAnimation 0.5s ease both - Appliquer l'animation
  • transform: translateY(-4px) scale(1.02) - Effets de hover performants
💡 Performance

Animer uniquement transform et opacity - ces propriétés sont optimisées par le GPU et n'entraînent pas de reflow. Éviter d'animer width, height, top, left.

🟡 Chapitre 3

JavaScript ES6+ - Interactivité

Le JavaScript côté client permet de rendre les pages interactives et dynamiques. Maîtriser ES6+, la manipulation du DOM et les requêtes asynchrones est la base du développement front-end.

Déclarations de variables - ES6+

Mot-cléPortéeRéassignableHoisting
varFonction✓Oui (à éviter)
letBloc✓Non (TDZ)
constBloc✗Non (TDZ)
💡 Règle pratique

Utiliser const par défaut. Passer à let uniquement si la valeur doit être réassignée. Ne jamais utiliser var.

Fonctions ES6+

  • Arrow function : const double = x => x * 2; - Syntaxe concise, pas de this propre
  • Paramètres par défaut : function saluer(nom = 'Utilisateur') { }
  • Template literals : `Bonjour ${nom}, tu as ${age} ans`
  • Destructuring : const { title, level } = formation;
  • Spread : const all = [...array1, ...array2];
  • Optional chaining : user?.address?.city - Évite les erreurs null

Manipulation du DOM

Le DOM (Document Object Model) est l'interface qui représente la page HTML sous forme d'arbre d'objets manipulables.

MéthodeRôle
document.getElementById('id')Sélectionne par ID (1 élément)
document.querySelector('.classe')Premier élément correspondant au sélecteur CSS
document.querySelectorAll('li')NodeList de tous les éléments correspondants
element.innerHTML = '...'Modifie le contenu HTML (attention XSS !)
element.textContent = '...'Modifie le texte (sûr contre XSS)
element.classList.add/remove/toggleModifie les classes CSS
element.setAttribute('aria-expanded', 'true')Modifie un attribut HTML

parent.appendChild(enfant)Ajoute un enfant dans le DOM

Événements

  • element.addEventListener('click', handler) - Attacher un événement
  • element.removeEventListener('click', handler) - Supprimer un événement
  • Délégation d'événement : écouter sur le parent et cibler avec event.target
  • event.preventDefault() - Annuler le comportement natif (ex: soumission de formulaire)
  • event.stopPropagation() - Empêcher la propagation vers les parents

Les événements courants : click, input, submit, keydown, scroll, resize, DOMContentLoaded.

AJAX - fetch() et async/await

AJAX (Asynchronous JavaScript and XML) permet de communiquer avec un serveur sans recharger la page. En ES6+, on utilise fetch() avec async/await.

  • fetch(url) retourne une Promise
  • response.ok - Vérifie si la réponse est un succès (status 200-299)
  • response.json() - Parse la réponse JSON (aussi une Promise)
  • Toujours entourer d'un try/catch pour gérer les erreurs réseau
ℹ Promesses vs async/await

Les deux sont équivalents. async/await est plus lisible pour les débutants. .then().catch() est plus compact pour les opérations chaînées simples. En pratique, préférer async/await.

Méthodes de tableaux ES6+ indispensables

.map() .filter() .reduce() .find() .forEach() .some() .every() [...new Set()] Array.from()
  • .filter() - Retourne un nouveau tableau avec seulement les éléments qui passent le test
  • .map() - Transforme chaque élément et retourne un nouveau tableau
  • [...new Set(array)] - Supprime les doublons d'un tableau

🟢 Chapitre 4

Accessibilité Web - WCAG & ARIA

L'accessibilité web permet aux personnes en situation de handicap d'utiliser les sites. Les WCAG 2.1 (Web Content Accessibility Guidelines) définissent 4 principes : Perceptible, Utilisable, Compréhensible, Robuste (POUR).

Les attributs ARIA essentiels

ARIA (Accessible Rich Internet Applications) complète le HTML sémantique là où il ne suffit pas.

AttributValeursUsage
aria-labelTexteÉtiquette pour les éléments sans texte visible
aria-labelledbyID d'un élémentRéférencer un titre existant
aria-describedbyID d'un élémentDescription complémentaire (aide/erreur)
aria-expandedtrue / falseÉtat ouvert/fermé d'un menu/accordéon
aria-hiddentrueMasquer aux technologies d'assistance
aria-livepolite / assertiveAnnonce les changements dynamiques
aria-currentpage / trueIndique l'élément actif dans une navigation
rolebutton, list, alert…Redéfinir le rôle sémantique

Checklist accessibilité

  • Utiliser des balises HTML sémantiques (nav, main, header, footer…)
  • Tous les éléments interactifs sont accessibles au clavier (Tab, Enter, Espace, Echap)
  • Le focus est visible et ne jamais faire outline: none sans alternative
  • Chaque image a un attribut alt (vide si décorative)
  • Ratio de contraste ≥ 4.5:1 pour le texte normal, ≥ 3:1 pour le texte large (WCAG AA)
  • Les formulaires ont des <label> liés aux <input>
  • La langue est déclarée avec lang="fr"
  • Les messages d'erreur sont annoncés avec aria-live
  • Pas d'informations transmises uniquement par la couleur
  • Éviter les liens "Cliquez ici" - utiliser des textes descriptifs
  • Ne jamais supprimer le style de focus sans le remplacer

🟣 Chapitre 5

Responsive Design

Le Responsive Web Design adapte la mise en page à tous les écrans. La meta viewport, les media queries, et les unités relatives sont les 3 piliers.

Breakpoints recommandés

NomBreakpointCibles typiques
Mobile< 480pxSmartphones (portrait)
Mobile large480px – 600pxGrands smartphones
Tablette600px – 900pxTablettes, petit laptop
Desktop900px – 1200pxLaptops standard
Large desktop> 1200pxGrands écrans

Approche Mobile-First

On écrit d'abord les styles pour mobile, puis on ajoute des media queries pour les plus grands écrans avec min-width.

💡 Mobile-first vs Desktop-first

Mobile-first (min-width) : meilleure performance sur mobile, approche progressive recommandée.
Desktop-first (max-width) : utile pour refactoriser un site existant desktop.

Unités relatives indispensables

  • % - Relatif au parent (largeurs fluides)
  • em - Relatif à la taille de police du parent
  • rem - Relatif à la taille de police racine (html) - Préférable
  • vw / vh - Relatif à la largeur/hauteur du viewport
  • clamp(min, val, max) - Valeur fluide entre deux bornes : font-size: clamp(1rem, 2.5vw, 2rem)

Images responsives

  • max-width: 100% sur toutes les images par défaut
  • <img srcset="..."> - Charger la bonne résolution selon l'écran
  • <picture> + <source media="..."> - Images différentes selon le viewport

Grille responsive automatique

La combinaison la plus puissante pour une grille qui s'adapte sans media query :

grid-template-columns: repeat(auto-fill, minmax(280px, 1fr))

Cela crée autant de colonnes que possible d'au moins 280px, et les étire pour remplir l'espace disponible.

🔴 Chapitre 6

Bonnes pratiques & Qualité

Un code de qualité professionnelle est lisible, maintenable, performant et sécurisé. Ces pratiques sont évaluées dans la grille EC01.

Organisation du code HTML/CSS/JS

  • Indentation cohérente (2 ou 4 espaces, au choix mais uniforme)
  • Noms de classes CSS descriptifs et cohérents (BEM, ou camelCase)
  • Commentaires expliquant le pourquoi, pas le quoi
  • Fonctions JS courtes avec un seul rôle (single responsibility)
  • Noms de variables et fonctions clairs : fetchFormations() pas fF()
  • Séparation des préoccupations : HTML structure, CSS présentation, JS comportement

Performance

  • Charger les scripts JS à la fin du <body> ou avec defer
  • Minifier CSS et JS en production
  • Compresser les images (WebP, AVIF)
  • Utiliser loading="lazy" sur les images hors viewport
  • Éviter les requêtes fetch redondantes - mettre en cache si possible
  • Animer avec transform et opacity uniquement (GPU)

SEO de base

  • <title> unique et descriptif (50-60 caractères)
  • <meta name="description"> (150-160 caractères)
  • Un seul <h1> par page, contenant le mot-clé principal
  • URLs propres et lisibles
  • Attributs alt descriptifs sur les images
  • Données structurées (JSON-LD) pour les moteurs de recherche

Sécurité front-end minimale

  • Ne jamais insérer de données utilisateur directement dans innerHTML - risque XSS
  • Utiliser textContent pour les textes non-HTML
  • Toujours valider les données côté serveur (la validation JS côté client seule ne suffit pas)
  • Ne jamais stocker de données sensibles dans localStorage
ℹ Niveaux d'importance pour l'examen EC01

La grille évalue : structure HTML5 et W3C (C1), qualité CSS et UX (C2), responsive (C3), JavaScript et ECMAScript (C4), AJAX/fetch (C5). Chaque critère vaut entre 0 et 5 points pour un total sur 55 points.
